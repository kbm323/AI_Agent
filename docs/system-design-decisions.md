# AI Virtual Entertainment Company — Interview-Driven Design Decisions

> Generated from Ouroboros interview session `interview_20260610_070847`
> Date: 2026-06-10

---

## Overview

**Goal:** Implement a meeting-first AI Virtual Entertainment Company operating system where a Hermes-based Coordinator dynamically routes user requests to role-specific AI workers, orchestrates multi-agent round-table discussions, builds consensus through structured validation, optionally delegates execution tasks to OpenClaw, and continuously evolves organizational knowledge through a four-layer memory architecture (Google Drive → Obsidian LLM Wiki → FTS5 Retrieval → Coordinator).

---

## Track 1: Multi-Model Worker Execution

**Original Issue:** Hermes `delegate_task` uses a single delegation model, but the design requires per-role different models.

**Decision:** opencode-go CLI direct invocation (primary) + tmux Hermes processes (persistent roles only).

**Details:**
- Coordinator calls `opencode-go --model <model> --context-file <packet>` directly
- Hermes `delegate_task` is NOT used for worker execution (no per-task profile override)
- Hermes Core modifications are avoided
- 90% general meetings: opencode-go CLI
- 10% special work: direct model invocation
- `delegate_task` core extension (Option A) is rejected

**Rationale:**
- opencode-go already supports multiple models (Kimi, DeepSeek, Qwen, GLM, MiniMax) via single interface
- Coordinator just passes model name as argument — simple implementation
- Hermes Core modification zero

## Track 2: Context Passing Between Rounds

**Decision:** Structured JSON packet generated by Coordinator per round. Raw originals preserved in memory/skills.

**Details:**
- Coordinator generates "meeting minutes packet" in JSON after each round
- Packet contents: per-role key arguments, evidence summary, rebuttal points, unresolved issues, numeric/reference data
- Next round subagent receives JSON packet as context injection
- Raw utterances stored in memory/skills only; round-to-round transfer uses packets only

**Design Principle:** Low token usage, zero information loss.

---

## Track 3: Routing

**Decision:** Qwen LLM primary classification + static routing table as fallback/cache.

**Details:**
- User question → Qwen LLM analyzes → determines required roles
- Static table (`routing_rules.yaml`) serves as fallback safety net
- Routing result: agenda_type, tags, risk_tags, required_roles, optional_roles, validator_required, codex_required
- Static table is guardrail, not primary router

**Rejected:** Pure static table (maintenance burden), embedding-based matching (complexity).

---

## Track 4: Validation Layer

**Decision:** GLM-5.1 default validator + conditional Codex GPT-5.5 dual validation.

**Escalation Triggers (GLM → Codex):**
- Budget/schedule related items
- Legal/copyright related items
- External publication/brand direction
- Code/automation/security/data loss risk
- Unstable consensus in meeting
- Latest-information-dependent items
- GLM returns non-pass verdict, confidence below threshold, or escalation recommendation

**Conflict Resolution (GLM vs Codex):**
- Technical judgment → Codex priority
- Data loss/operational change/irreversible → User escalation
- Legal/budget/brand/external → User escalation
- Factual issues → Re-validate after search/verification

**Out of Scope:** GLM solo re-validation is NOT considered dual validation.

## Track 5: CLI Contract (Coordinator ↔ opencode-go)

**Decision:** File-based JSON I/O.

**Context Injection:**
- `--prompt` inline is FORBIDDEN (ARG_MAX, escaping issues)
- Primary: `--context-file` argument
- Fallback: stdin redirection

**Output Collection:**
- Structured output flag if available
- Otherwise: prompt enforces JSON code block at stdout end
- Coordinator extracts last JSON block → parse → schema validation

**Failure Handling:**
- Signals: exit code ≠ 0, timeout, stderr error pattern, JSON schema validation failure
- Step 1: Same model retry × 1
- Step 2: Fallback model
- Step 3 (2 failures): `agent_failed` record + validation escalation trigger

**Design Principle:** opencode-go CLI treated as structured worker, not non-deterministic NLG tool. All I/O filed by meeting_id for reproducibility, debugging, knowledge accumulation.

## Track 6: System-Level Error Recovery

**A. Deadlock (Consensus Failure):**
- Default max 3 rounds + 1 tie-break round
- Still no consensus → branch by item type:
  - General planning: Coordinator writes chairperson proposal
  - Technical conflict: GLM validation → Codex priority
  - Budget/legal/brand/external/irreversible: User escalation
- Infinite meetings forbidden

**B. Provider Failure:**
- Required roles: retry with fallback model
- Optional roles: skip allowed
- Validator failure: GLM fallback → Codex validation → User escalation

**C. Quorum (Partial Response):**
- Minimum quorum: 2/3, 3/4, 3/5, 4/6, 4/7
- Required roles + validator response MUST be secured regardless of quorum
- Not secured → retry round or escalation

**D. Coordinator Failure:**
- All meetings file-saved by meeting_id
- Resume from last completed step per manifest.json
- Only damaged/incomplete agent calls re-executed
- Design Principle: All failures recorded in meeting manifest. Silent fail forbidden.

## Track 7: OpenClaw Delegation Interface

**Decision:** OpenClaw = Tool-use Executor (Option B). NOT alternative orchestrator.

**Role Boundary:**
- Hermes Coordinator: thinking, meeting, consensus, validation, routing
- opencode-go Workers: per-role opinion generation
- OpenClaw Workers: actual execution and artifact creation

**Delegation Targets (OpenClaw):**
- Code generation and large-scale code modification
- Multi-file editing
- Git commit/branch/PR creation
- Build/test/deploy automation
- CLI-based automation
- Browser/external tool manipulation
- Long-running execution workflows

**Delegation Exclusions (NOT OpenClaw):**
- Idea generation, meetings/discussions, opinion gathering, validation/rebuttal, legal/brand/strategy decisions, final decision-making

**Interface Contract:**
- Same file-based JSON contract as opencode-go
- Input: meeting_id + Action Plan + Context Packet (JSON)
- Output: structured JSON (status, summary, artifacts, errors, needs_review)
- Coordinator collects results → validation layer → final response integration

**Response Ownership:**
- OpenClaw does NOT respond directly to users
- User interface: Coordinator single window always

**Design Principles:**
- Execution Separation: thinking vs doing
- Pluggable execution layer: OpenClaw replaceable without touching Coordinator

## Track 8: Concurrent Meeting Handling

**Decision:** Priority Queue + Limited Parallelism.

**Priority Levels:**
- P0: Incident response, data loss, security, urgent fixes
- P1: In-progress project decisions, legal, budget, external publication
- P2: General production, automation, development
- P3: Ideas, brainstorming
- Same priority: FIFO

**Resource Limits:**
- max_concurrent_meetings: 2
- max_concurrent_opencode_calls: 4
- max_agents_per_meeting: 7
- max_rounds_per_meeting: 3
- max_tiebreak_rounds: 1
- P0: can suspend lower-priority meetings
- P2/P3: enter queue

**Session Isolation:**
- Directory: `meetings/{meeting_id}/`
- Data: input, routing, round outputs, consensus, validation, final
- Shared: `runtime/` global state files only

**User Feedback:**
- Display: queue position, priority, current phase, completed agents, fallback usage, cancel availability
- No ETA — phase-based progress only
- Intermediate preview: must show "pre-validation" label
- Cancel: `/cancel {meeting_id}` → manifest records cancelled, outputs preserved

## Track 9: Knowledge Accumulation

**Decision:** 4-layer knowledge system.

**Layer 0 — Google Drive (External Knowledge Repository):**
- Source storage: PDF, SRT, GitHub ZIP/README, images, reference materials, project originals
- Immutable — never modify or delete originals

**Layer 1 — Obsidian LLM Wiki:**
- Structure: Wiki/, Meetings/, Decisions/, Projects/, Agent_Specs/, Summaries/, System_Reviews/
- AI-generated summaries, links, wikis, decision documents only
- NOT originals — Obsidian is the AI's working knowledge layer

**Layer 2 — context-mode FTS5 (Search Engine):**
- Markdown FTS, Decision Log search, Meeting History search, Wiki search
- Vector DB is future optional extension

**Layer 3 — Hermes Coordinator (Dynamic Retrieval):**
- Coordinator-driven dynamic retrieval: analyze meeting purpose → search relevant knowledge → Context Packet
- Past knowledge NOT injected into all meetings by default
- User explicit request: force reference specific past meetings/documents

**Knowledge Lifecycle:**
- Immutable: Google Drive Source, Meeting Records, Decision Logs (append-only, permanent)
- Living: Wiki, Project Summary, Knowledge Summary (AI continuously updates)
- Stale conclusions: NOT deleted; metadata managed (status: superseded, superseded_by, reason)
- Auto-summarization: Weekly, Monthly, Quarterly digests

**Meta-Cognition:**
- Collected metrics: Deadlock frequency, Validator disagreement rate, Fallback usage, per-model success/failure, per-role bottlenecks, escalation frequency
- Self-Reflection Report: Coordinator generates periodic system operations report + improvement proposals
- CRITICAL: Meta-cognition results NOT auto-applied — user approval required for policy changes

## Track A: Discord Integration Surface

**Meeting Initiation:**
- Primary: Slash Command (`/meeting agenda:"..."`)
- Secondary: Bot Mention (`@AI_Company ...`)
- Rejected: auto-processing all channel messages, polling general chat

**Coordinator Intake:**
- Hermes existing Discord Gateway utilized
- Adapter Layer only — no separate gateway implementation

**Response Delivery:**
- Primary: Discord Thread from original message = meeting space
- Thread contains: meeting start → per-role utterances → validation → execution results → final conclusion
- Secondary: #회의-결과 or #일일-브리핑 summary post
- DM: only for approval requests, private info

**Follow-up Handling:**
- Default: same Thread follow-up extends existing meeting_id
- New meeting_id: `/meeting new`, complete topic change, team change, final/locked state, long time elapsed
- meeting_id tracks: user_id, channel_id, thread_id

**Discord Persona Architecture — Hybrid Bot/Webhook Model:**
- User-approved direction: use the hybrid model, not webhook-only and not one real bot per specialist.
- Persistent team-leader bots (6-7): 👔Coordinator, 🎬ContentPD, 🎨ArtDirector, ⚙️TechDirector, 📈MarketingLead, 🛡️Validator, 🛠️ExecutionTeam/OpenClaw.
- Team-leader bots are real Discord bots/users from the user's point of view: they can be mentioned directly (for example `@TechDirector ...`), can own visible team-level responses, and can participate in meeting threads.
- Internal specialists are not real Discord bots by default; they exist as opencode-go/Hermes workers and are represented through the relevant team leader.
- Optional specialist appearance uses Discord webhook rendering inside the meeting thread (custom username/avatar per message) for roles like Live2D rigger, 3D modeler, Unity engineer, motion designer, QA analyst.
- Webhook specialists are presentation-only: they do not receive mentions, DMs, slash commands, or presence. If a specialist must be directly mentionable, promote that role to a persistent bot explicitly.
- Coordinator owns `/meeting`, thread creation, meeting_id routing, state transitions, and final cross-posts. Team-leader bots own role-specific visible utterances. Webhook specialists only render internal worker outputs.

**Design Principles:**
- Mentionable roles require persistent bots; presentation-only specialist voices use webhooks.
- Persona over Model: Discord exposes roles, not models.
- Persistent Leaders, Dynamic Specialists.
- Single User Interface: one AI company, many internal personas.

## Track B: Meeting Orchestration State Machine

**Normal States:**
created → queued → routing → context_retrieval → in_meeting → consensus_building → validating → executing(optional) → finalizing → completed

**Exception States:**
paused, deadlocked, escalated, cancelled, failed

**State Transitions (Coordinator only):**
- created: `/meeting` or mention received
- queued: queue slot secured
- routing: routing complete
- context_retrieval: Context Packet generated
- in_meeting: minimum quorum + required role responses secured
- consensus_building: consensus draft created
- validating: consensus sent to validation
- executing: if execution needed (can re-enter validating after)
- finalizing: final report generation

**Validation Failure Branches:**
- minor_issue → fix
- missing_info → re-retrieve
- disagreement → re-meeting
- high_risk → Codex 2nd validation or escalation

**State Ownership:**
- Coordinator owns: meeting_id, current_state, priority, participants, round_count, quorum, validation, execution, final status
- Workers return events: worker_completed, worker_failed, validation_result, execution_result
- Storage: `meetings/{meeting_id}/manifest.json` — Source of Truth

**Error Recovery by State:**
- Worker timeout → retry 1 → fallback → agent_failed
- Provider failure → fallback model/provider
- Quorum unmet → paused → wait/fallback
- Deadlock → tie-break 1 → chairperson proposal/escalation
- GLM failure → GLM fallback/Codex validation
- GLM/Codex conflict → high-risk escalation, normal → re-meeting
- OpenClaw failure → retry → record failure → validation/escalation
- P0 preemption → lower paused, resume from manifest after P0
- User cancel → cancelled record, output preserved
- Coordinator crash → resume from manifest.completed_step

**Design Principle:** State is centralized, execution is distributed. Every state transition persisted before next external call.

## Track C: Agent Persona Specification

**File Format:** Hybrid — `agent.yaml` (structured execution config) + `persona.md` (persona & operations guide).

**Required Fields (agent.yaml):**
- role_id, display_name, team, role_type
- model: provider, name, fallback, temperature, max_tokens
- expertise_tags
- routing: required_for_tags, optional_for_tags
- meeting: quorum_weight, response_schema
- discord: webhook_name, avatar_key, persistent_persona
- memory: memory_scope, write_to_obsidian
- version: spec_version, status
- Validator/Executor: validation/execution sections

**Storage:**
- Execution: `agents/{team}/{role_id}/agent.yaml` + `persona.md` (Git repo)
- Obsidian: Agent_Specs documentation copy only (not execution originals)

**Versioning:**
- Git version control + spec_version + git_commit
- Meeting start: snapshot spec_version + git_commit in manifest.json
- In-progress meeting: unaffected by spec changes
- Deprecation: `status: deprecated`, never deleted
- Emergency critical fixes only: stop current meeting, restart with new meeting_id

**Design Principle:** Persona is configuration, not code. Every output traceable to spec version.

## Track D: Context Packet Exact Schema

**Physical Structure (Dual):**
- `round_N/context_packet_{role_id}.json` — Worker execution context
- `meeting_context.json` — cumulative meeting summary
- `raw_outputs/` — original outputs preserved (reproducibility, audit)

**Round-by-Round Content:**
- Round 1: user_request, agenda, meeting_goal, role_instruction, constraints, retrieved_knowledge, required_output_schema (independent opinion)
- Round 2: previous_round_summary, consensus_so_far, known_conflicts, relevant_agent_outputs, risks, missing_info (conflict resolution)
- Round 3: candidate_consensus, unresolved_conflicts, decision_options (final convergence)
- Tie-break: deadlock_summary, positions_by_side, cost_of_each_option

**Worker Delivery:**
- Primary: `--context-file` path
- Fallback: stdin redirection
- Forbidden: `--prompt` inline
- Coordinator pre-filters by target_role (Worker does NOT self-filter)

**Size Limits:**
- Normal worker: 12k tokens
- Validator: 20k tokens
- Codex: 30k tokens
- Priority preservation: user_request, meeting_goal, role_instruction, constraints, known_conflicts, candidate_consensus, validation findings
- First to compress/exclude: long raw quotes, repetitive utterances, low relevance docs, previous round full originals
- All truncation recorded in `omitted_sections`
- Long meetings: rolling summary, continuation meeting when scope grows

**Design Principle:** Context is curated, not dumped. Raw data preserved for audit; Workers receive only role-relevant, bounded, versioned Context Packet.

## Track E: Internal Validator Mechanism

**Validation Scope (5 areas):**
- requirements_fit, logical_consistency, factual_grounding, feasibility, risk_policy
- Sequence: Requirements Check → Logic/Feasibility Check → Risk/Escalation Check

**Scoring:**
- 5-level verdict: pass, conditional_pass, revision_required, escalate, fail
- Per-area 0~1 score → overall_score
- overall_score >= 0.85 + no high risk → pass
- >= 0.75 + minor issues only → conditional_pass
- 0.60~0.75 → revision_required
- high risk → escalate
- < 0.60 → fail
- Critical AC failure → revision_required/escalate regardless of score

**Output Schema:**
- Required: verdict, overall_score, confidence, scores(per area), risk_level, issues[], missing_requirements, codex_escalation, user_escalation, recommended_next_state, summary

**Codex 2nd Validation Triggers:**
- codex_escalation.required = true
- risk_level = high
- verdict = escalate
- Code/automation/security/data loss issues
- GLM confidence < 0.75
- GLM-consensus core judgment conflict

**Conflict Resolution (GLM vs Codex):**
- Technical/security/data → Codex priority
- Legal/budget/brand → User escalation

**Cycle Prevention:**
- Max validation: 2 times
- Max re-meeting after validation: 1 time
- Max Codex validation: 1 time
- Same issue category 2 repeats → auto escalation
- Validator output schema validation failure → repair 1 → GLM fallback/Codex

**Design Principle:** Validation is structured, bounded, auditable. Validator emits issues+scores+state; Coordinator owns final transition.

## Track F: Multi-Model Worker Deployment & Routing

**Execution Mechanism:**
- Adopted: opencode-go CLI direct invocation
- Rejected: delegate_task (no profile support), config dynamic switching (race condition), core extension (principle conflict)

**Routing:**
- Qwen LLM primary classification
- routing_rules.yaml: fallback/guardrail only
- Output: agenda_type, tags, risk_tags, required_roles, optional_roles, validator_required, codex_required

**Parallelism (Hybrid):**
- Parallel: Round 1, Round 2 independent opinion gathering
- Sequential: Consensus writing, Validation, Codex 2nd validation, OpenClaw execution, post-execution re-validation
- Worker failure: does NOT halt entire meeting; Coordinator judges by quorum + required_roles

**Fallback & Retry:**
- Sequence: Retry × 1 → Fallback × 1 → Quorum reassessment → Escalation/Fail
- Timeouts: Normal 120s, Required/Validator 180s, Codex 240s
- Required Worker: retry + fallback required
- Optional Worker: skip allowed
- GLM failure → GLM-5 fallback or Codex promotion
- Codex failure → User escalation
- Fallback usage + failure reason recorded in manifest

## Track G-1: OpenClaw Execution Model

**Hybrid — Short sync, Long async:**
- ≤30s (code_patch, git_diff, short run_tests): sync call
- >30s (build_project, create_pr, automation): async job
- High-risk (deploy, PR create, external publish, bulk file change, data delete/migrate): async + checkpoint + human approval
- Sync timeout → async job conversion or fallback

## Track G-2: OpenClaw Async Protocol

**Q1 — Sync vs Async:** Hybrid (above)

**Q2 — Async Notification:** Hybrid (Callback primary + File Watch fallback + Polling recovery)
- Primary: Callback/Webhook — OpenClaw sends execution_result to Coordinator callback endpoint
- Fallback: File Watch — execution_result.json creation detection
- Recovery: Polling — health check only, status endpoint query when callback missing

**Q3 — Runtime Intervention:** Cancel-Only + Semantic Retune
- Cancel: user stop, timeout, P0 preemption, risk detection → direct call
- Pause: cancel + preserve checkpoint/result → paused state
- Retune: cancel + new execution_id + modified execution_packet
- Human approval: pre-execution wait state; mid-execution discovery → cancel → approval state
- All cancel/retune: manifest records original_execution_id, new_execution_id, reason, preserved_artifacts

**Design Principle:** External execution immutable once started. Scope/parameter change = new auditable execution, not mutation of running process.

## Summary of Design Principles

1. **State is centralized, execution is distributed** — Coordinator owns state machine; workers emit events
2. **Persona over Model** — Discord exposes roles, not models; internal model swaps transparent
3. **Context is curated, not dumped** — Workers receive role-relevant bounded packets; raw data preserved for audit
4. **Execution is permissioned, auditable, reversible** — OpenClaw acts; Coordinator decides; Validator checks; User approves irreversible
5. **Validation is structured, bounded, auditable** — Validator emits issues+scores; Coordinator owns transition
6. **Knowledge Separation** — Source=Fact, Wiki=Interpretation, Decision=Choice, Summary=Compression
7. **Retrieval First** — Past knowledge injected only when relevant, not by default
8. **Human Approval** — Meta-cognition proposals require user approval; never auto-applied
9. **Silent fail forbidden** — All failures recorded in manifest; final response declares missing roles, fallback usage, escalation status
10. **Hermes Core minimal modification** — All extensions via external CLI calls, profiles, or adapter layers
